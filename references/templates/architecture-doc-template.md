## What Makes a Good Architecture Doc

1. **Captures the "why"** — anyone can read the code to see "what", but the reasoning behind
   choices gets lost without documentation
2. **Stays current** — include a note at the top encouraging updates when the architecture changes
3. **Links to code** — reference actual file paths so readers can jump to the implementation
4. **Diagrams** — a mermaid diagram is worth a thousand words of explanation
5. **Decision records** — the table format makes it easy to revisit decisions later
